## 一、总体方案：复合流水线

```
输入文本 (str)
    │
    ▼
[数据预处理] ──→ 清洗 + Tokenize
    │
    ▼
[RoBERTa分类器] ──→ label ∈ {0, 1}
    │
    ▼
[Captum IG 归因] ──→ evidence_tokens (Top-K)
    │
    ▼
[Prompt + CLAW LLM] ──→ explanation (str)
    │
    ▼
输出: {"label": int, "confidence": float, "explanation": str, "evidence": dict, "event": int}
```

**分类器**负责拿满准确率（15分），**解释器**负责拿满可解释性（15分），**工程与报告**负责拿满代码可运行与文档（25分）和报告（30分）。

## 二、荣：数据与 Baseline

### 2.1 职责
建立数据流水线，提供可靠的数据洞察，并设置性能基准线（Baseline），让刘的深度学习模型有明确优化目标。

### 2.2 技术细节

#### 数据探索（EDA）
- 读取 `train.csv` / `val.csv`，检查缺失值、重复行、`label` 分布、`event` 类别数。
- 统计文本长度分布（分词前/后），为后续 `max_length` 设置提供依据。
- 用 `wordcloud` 或 `matplotlib` 分别绘制 rumor / non-rumor 的高频词云。

#### 数据清洗与加载
- 清洗规则（建议）：
  - 去除 URL（正则 `http\S+`）
  - 去除 `@用户名` 与 `#话题#`（保留或去除均可，需统一并在报告中说明）
  - 去除多余空格、换行符
- 封装 `RumorDataset(torch.utils.data.Dataset)`：
  ```python
  class RumorDataset(Dataset):
      def __init__(self, csv_path, tokenizer, max_len=128):
          ...
      def __getitem__(self, idx):
          return {"input_ids": ..., "attention_mask": ..., "label": ...}
  ```
- 封装 `get_dataloaders(train_csv, val_csv, batch_size, tokenizer)`，支持 `num_workers`。

#### Baseline
- **Baseline-1**：TF-IDF (max_features=5000) + Logistic Regression
- **Baseline-2**：TextCNN 或 BiLSTM + GloVe
- 统一评估指标：`accuracy`, `precision`, `recall`, `f1-score`
- 所有 Baseline 结果保存到 `results/baseline_results.json`

### 2.3 交付物
| 文件 | 说明 |
|------|------|
| `data_loader.py` | `RumorDataset`, `get_dataloaders`, 清洗函数 |
| `eda.ipynb` | 数据探索与可视化 |
| `baseline.py` | TF-IDF/LR 与 TextCNN 训练脚本 |
| `results/baseline_results.json` | 基准指标 |

### 2.4 接口契约（供刘/钱调用）

## 三、刘：核心分类模型

### 3.1 职责
实现并调优谣言分类的核心深度学习模型，在 `val.csv` 上取得尽可能高的准确率。

### 3.2 技术细节

#### 模型选型
- 主模型：`roberta-base`（HuggingFace `cardiffnlp/twitter-roberta-base` 更佳，若不可用则通用 `roberta-base`）
- 对比实验（可选）：`bert-base-uncased`, `deberta-v3-small`
- 头部结构：在 `[CLS]` 后接 `Dropout(0.1)` + `Linear(hidden_size, 2)`

#### 训练策略
- 优化器：`AdamW`
- 学习率：`2e-5`，用 `get_linear_schedule_with_warmup`（warmup比例 0.1）
- 早停（EarlyStopping）：监控 `val_f1`，耐心 `patience=3`
- 保存策略：每个 epoch 保存 checkpoint，只保留最佳模型
- 冻结策略：前 6 层冻结 + 后 6 层微调（显存不足时采用）

#### 评估与输出
- 输出 `classification_report`、混淆矩阵热力图
- 错误分析：随机抽取 20 条错误样本，保存到 `results/error_cases.json`
- 最佳模型保存路径：`checkpoints/best_model.pt`（或 HuggingFace 格式目录）

### 3.3 交付物
| 文件 | 说明 |
|------|------|
| `model.py` | `RumorClassifier` 模型定义 |
| `train.py` | 训练脚本，支持命令行参数 `--epochs`, `--lr`, `--batch_size` |
| `evaluate.py` | 在 val.csv 上评估，打印指标并生成混淆矩阵图 |
| `checkpoints/` | 最佳模型权重与 tokenizer |
| `results/training_curves.png` | loss / acc / f1 曲线 |

## 四、钱：可解释性与 LLM 接口

### 4.1 职责
为分类结果提供可信、可读的解释，并参与模型调试、数据清洗与安全鲁棒性分析。使用 Captum 提取关键证据，通过 CLAW API 生成自然语言解释，同时对训练数据中的潜在错标 / 投毒样本进行排查。

### 4.2 技术细节

#### Captum 归因
- 使用 `captum.attr.IntegratedGradients`
- 在 embedding 层计算归因值，避免直接对离散 token id 求梯度
- 兼容 RoBERTa / BERT / DeBERTa 等不同 backbone 的 embedding 层命名
- 提取对「预测类别」贡献最高的 Top-K tokens（K=5~10，过滤 `[CLS]`, `[SEP]`, `[PAD]` 及标点）
- 实现 `get_evidence_both_sides(text, predicted_label, top_k=5)`，同时返回支持预测标签和支持对立标签的证据，供 LLM 生成平衡解释
- 对每个 token 给出归一化贡献分数（百分比）

#### 解释可视化（调试与报告用）
- 使用 `captum.attr.visualization` 生成单条文本的 token 归因可视化：
  - 正向贡献使用绿色，负向贡献使用红色
  - 同时保存 HTML 高亮图和 JSON 原始归因数据
  - 输出目录：`results/explanation_examples/`
- 可视化用于报告展示和人工核查：确认模型证据词是否真实出现在文本中、是否与预测标签一致
- 命令行用法：
  ```bash
  python visualize_explanation.py \
    --model_dir checkpoints \
    --text "Breaking: unconfirmed reports say the bridge collapsed." \
    --prefix bridge_example
  ```
- 输出文件：
  - `results/explanation_examples/bridge_example_attribution.html`
  - `results/explanation_examples/bridge_example_attribution.json`

#### LLM 解释生成（CLAW API）
- **Prompt 设计**（含正反两面论述）：
  - System: 要求 LLM 提供平衡解释，包含 (1) 预测标签与置信度 (2) 支持证据 (3) 反方证据 (4) 引用具体词汇
  - User: 包含 tweet 文本、event 类别、预测标签、置信度、支持证据 token、反方证据 token
  - 温度 0.3，max_tokens 512，英文输出，3-5 句话
- 封装 `llm_client.py`：
  - API 地址、Key 从环境变量读取（`CLAW_API_KEY`, `CLAW_BASE_URL`）
  - 默认模型 `minimax`，端点 `https://models.sjtu.edu.cn/api/chat/completions`
  - 含重试机制（最大3次，指数退避）
  - 超时处理（timeout=30s）
  - 异常降级：若 API 失败，返回模板解释（含正反两面）

#### 模型调试与解释质量改进
- 联调 `explainer.py`、`llm_client.py`、`inference.py`，保证分类 → 归因 → 解释的端到端流水线稳定运行
- 修复和验证不同 backbone 加载时的兼容性问题，配合 `model_config.json` 自动识别训练时使用的模型名称
- 对 DeBERTa、roberta-large 等模型实验结果进行分析，确认当前小数据集下 `roberta-base` 是性价比最高的默认模型

#### 数据投毒 / 错标样本排查
- 从安全鲁棒性角度检查训练集是否存在“语义相近但 label 相反”的可疑样本
- 使用字符级 TF-IDF + cosine similarity 筛选相似度 `>= 0.70` 且标签冲突的近重复样本对
- 输出人工核查表 `results/train_label_conflict_pairs_similarity_ge_0_70.csv`
- 对语义重复但标签不同的样本进行人工核查和清洗，更新 `train.csv`
- 在 `Project.md` 中记录数据投毒 / 错标排查方法、发现和处理结论

#### 对抗攻击鲁棒性分析
- 参与输入级字符扰动攻击测试设计，如 `o/O → 0`、`l/L → 1`、扩展 leetspeak 替换等
- 通过 Accuracy Drop、Flip Rate、Attack Success Rate 等指标评估现有模型在扰动输入下的脆弱性
- 相关结果用于报告中的安全分析和防护讨论

#### 人工抽查
- 从 val.csv 中随机选 30 条，人工核对「解释是否与标签一致、证据是否真实出现在文本中」。
- 记录不一致案例，反馈给钱/刘调优模型或 Prompt。

- `inference.py` 支持三种模式：

## 七、风险与应对

| 风险 | 责任人 | 应对 |
|------|--------|------|
| CLAW API 限额/延迟 | C | 实现模板降级；批量推理时增加 `sleep(0.5)`；提前缓存结果 |
| 显存不足（< 8G） | B | 换 `roberta-base`；开 `fp16`；减小 `batch_size` 到 16 或 8 |
| 解释与标签矛盾 | C | 人工抽查30条，调整Prompt要求「解释必须与证据词一致」；Prompt加入正反两面论述增强robustness |
| 数据集泄露/重复 | A | EDA阶段检查train与val的文本交集，若有重复需清洗并报告 |
| 环境冲突 | D | `requirements.txt` 固定主版本；推荐统一使用 Python 3.10 |
| Windows 下 torch + pandas 导入导致 segfault | D | **已修复**：确保 `import pandas` 在 `import torch` 之前执行 |
| rumor 类假阴性过多 | B | **已修复**：使用 `compute_class_weight('balanced')` 加权 CrossEntropyLoss，提高 rumor 类惩罚 |
| API Key 明文泄露 | D | **已修复**：移除代码中硬编码的 `sk-` fallback；统一使用 `.env` 文件 + `python-dotenv`；`.gitignore` 排除 `.env`；未配置时自动模板降级 |
