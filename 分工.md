## 一、总体方案：复合流水线

```
输入文本 (str)
    │
    ▼
[数据预处理] ──→ 清洗 + Tokenize
    │
    ▼
[RoBERTa分类器] ──→ label ∈ {0, 1}
    │
    ▼
[Captum IG 归因] ──→ evidence_tokens (Top-K)
    │
    ▼
[Prompt + CLAW LLM] ──→ explanation (str)
    │
    ▼
输出: {"label": int, "confidence": float, "explanation": str, "evidence": dict, "event": int}
```

**分类器**负责拿满准确率（15分），**解释器**负责拿满可解释性（15分），**工程与报告**负责拿满代码可运行与文档（25分）和报告（30分）。

## 二、荣：数据与 Baseline

### 2.1 职责
负责训练 / 验证数据的读取、清洗、封装与基础统计分析，为后续 RoBERTa 分类器训练、评估和可解释性模块提供稳定的数据输入接口。同时提供数据分布洞察，帮助确定 `max_len`、batch size、类别权重等关键训练参数。

### 2.2 技术细节

#### 数据探索（EDA）
- 读取 `train.csv` / `val.csv`，确认数据集字段为 `id`, `text`, `label`, `event`。
- 检查训练集和验证集的样本数量、标签分布、事件类别分布，为后续类别加权训练提供依据。
- 统计文本长度分布，支持将模型最大长度从 128 调整为 256，减少长推文截断造成的信息损失。
- 观察 rumor / non-rumor 样本的高频词、事件关键词和 URL / hashtag 等社交媒体特征，为错误分析和报告提供数据背景。

#### 数据清洗与加载
- 实现 `clean_text(text)`：
  - 去除 URL（正则 `http\S+`），避免模型过度依赖无语义链接；
  - 合并多余空白、换行符；
  - 保留原始推文中的事件关键词、hashtag 等可能有判别价值的信息。
- 封装 `RumorDataset(torch.utils.data.Dataset)`：
  - 读取 CSV 并生成 `text_clean`；
  - 调用 HuggingFace tokenizer 生成 `input_ids` 和 `attention_mask`；
  - 返回 `label`、`raw_text`、`event`，支持训练、评估和解释模块复用。
- 封装 `get_dataloaders(train_csv, val_csv, tokenizer, batch_size, max_len, num_workers)`：
  - 训练集 `shuffle=True`；
  - 验证集 `shuffle=False`；
  - 支持 `max_len=256` 和多进程加载。

#### Baseline 与数据支撑
- 提供传统文本分类 baseline 的设计思路：TF-IDF + Logistic Regression / Linear SVM，用于与深度学习模型进行性能对照。
- 统一评估指标口径：`accuracy`, `precision`, `recall`, `f1-score`。
- 为刘的训练模块提供类别分布数据，支持 `compute_class_weight('balanced')` 计算类别权重，缓解 rumor 类假阴性过多的问题。

## 三、刘：核心分类模型

### 3.1 职责
负责谣言检测核心分类模型的实现、训练、调参与评估，在 `val.csv` 上尽可能提高二分类准确率和 rumor 类召回率，并为解释模块和系统集成模块提供稳定的模型加载与预测接口。

### 3.2 技术细节

#### 模型选型与结构
- 主模型确定为 `roberta-base`：在当前小规模数据集上性能和训练成本平衡最好。
- 模型结构：`AutoModel` backbone + `[CLS]` token 向量 + `Dropout(0.1)` + `Linear(hidden_size, 2)`。
- 支持通过 `--model_name` 切换 backbone，便于实验 DeBERTa、roberta-large 等模型。
- 在 `src/model.py` 中实现 `RumorClassifier`，统一返回 `{"logits": logits}` 或 `{"loss": loss, "logits": logits}`。
- 针对部分模型权重 dtype 不一致问题，在初始化后统一 `self.to(torch.float32)`。

#### 训练策略
- 优化器：`AdamW`，默认学习率 `2e-5`，默认 `weight_decay=0.01`。
- 学习率调度：`get_linear_schedule_with_warmup`，warmup steps 为总步数的 10%。
- 损失函数：使用 `compute_class_weight('balanced')` 计算类别权重，并传入加权 `CrossEntropyLoss`，重点缓解 rumor 类假阴性问题。
- 输入长度：默认 `max_len=256`，减少长推文截断。
- 梯度裁剪：`clip_grad_norm_(max_norm=1.0)`，提升训练稳定性。
- EarlyStopping：通过 `--patience` 控制验证集准确率长期不提升时提前停止。
- 梯度累积：通过 `--accumulation_steps` 支持显存受限时模拟更大 batch。
- 最佳模型按验证集准确率保存到 `checkpoints/best_model.pt`。

#### 模型配置与加载
- 训练结束后保存：
  - `best_model.pt`
  - tokenizer 文件
  - `training_history.json`
  - `model_config.json`
- `model_config.json` 记录训练时使用的 backbone 名称，供 `evaluate.py`、`inference.py`、`test_sample.py` 自动识别，避免更换模型后维度不匹配。
- 提供 `load_model_from_path(model_dir, model_name=None)`，支持本地 checkpoint 加载。
- 支持 HuggingFace 模型路径，例如 `charchar2333/Rumor-detection`，便于最终部署和助教复现。

#### 评估与错误分析
- `src/evaluate.py` 输出：Accuracy、F1、classification report、confusion matrix。
- 保存评估结果到：
  - `results/evaluation_results.json`
  - `results/error_cases.json`
  - `results/confusion_matrix.png`
- 当前主模型结果：`roberta-base + max_len=256 + 类别加权`，`val_acc=87.78%`，rumor recall 达 `90.86%`。
- 对错误样本进行分析，发现主要问题是假阴性：模型容易将事实性、制度性、法律分析式的谣言文本判为非谣言。

#### 模型对比实验
- 尝试 `microsoft/deberta-v3-base`，最高约 82%，表现不佳，判断为小数据集下过拟合 / 学习不充分。
- 尝试 `roberta-large`，最高约 88.78%，但 train loss 极低而 val loss 明显升高，严重过拟合，性价比低。
- 最终决策：默认模型回退并固定为 `roberta-base`。

## 四、钱：可解释性与 LLM 接口

### 4.1 职责
为分类结果提供可信、可读的解释，并参与模型调试、数据清洗与安全鲁棒性分析。使用 Captum 提取关键证据，通过 CLAW API 生成自然语言解释，同时对训练数据中的潜在错标 / 投毒样本进行排查。

### 4.2 技术细节

#### Captum 归因
- 使用 `captum.attr.IntegratedGradients`
- 在 embedding 层计算归因值，避免直接对离散 token id 求梯度
- 兼容 RoBERTa / BERT / DeBERTa 等不同 backbone 的 embedding 层命名
- 提取对「预测类别」贡献最高的 Top-K tokens（K=5~10，过滤 `[CLS]`, `[SEP]`, `[PAD]` 及标点）
- 实现 `get_evidence_both_sides(text, predicted_label, top_k=5)`，同时返回支持预测标签和支持对立标签的证据，供 LLM 生成平衡解释
- 对每个 token 给出归一化贡献分数（百分比）

#### 解释可视化（调试与报告用）
- 使用 `captum.attr.visualization` 生成单条文本的 token 归因可视化：
  - 正向贡献使用绿色，负向贡献使用红色
  - 同时保存 HTML 高亮图和 JSON 原始归因数据
  - 输出目录：`results/explanation_examples/`
- 可视化用于报告展示和人工核查：确认模型证据词是否真实出现在文本中、是否与预测标签一致
- 命令行用法：
  ```bash
  python visualize_explanation.py \
    --model_dir checkpoints \
    --text "Breaking: unconfirmed reports say the bridge collapsed." \
    --prefix bridge_example
  ```
- 输出文件：
  - `results/explanation_examples/bridge_example_attribution.html`
  - `results/explanation_examples/bridge_example_attribution.json`

#### LLM 解释生成（CLAW API）
- **Prompt 设计**（含正反两面论述）：
  - System: 要求 LLM 提供平衡解释，包含 (1) 预测标签与置信度 (2) 支持证据 (3) 反方证据 (4) 引用具体词汇
  - User: 包含 tweet 文本、event 类别、预测标签、置信度、支持证据 token、反方证据 token
  - 温度 0.3，max_tokens 512，英文输出，3-5 句话
- 封装 `llm_client.py`：
  - API 地址、Key 从环境变量读取（`CLAW_API_KEY`, `CLAW_BASE_URL`）
  - 默认模型 `minimax`，端点 `https://models.sjtu.edu.cn/api/chat/completions`
  - 含重试机制（最大3次，指数退避）
  - 超时处理（timeout=30s）
  - 异常降级：若 API 失败，返回模板解释（含正反两面）

#### 模型调试与解释质量改进
- 联调 `explainer.py`、`llm_client.py`、`inference.py`，保证分类 → 归因 → 解释的端到端流水线稳定运行
- 修复和验证不同 backbone 加载时的兼容性问题，配合 `model_config.json` 自动识别训练时使用的模型名称
- 对 DeBERTa、roberta-large 等模型实验结果进行分析，确认当前小数据集下 `roberta-base` 是性价比最高的默认模型

#### 数据投毒 / 错标样本排查
- 从安全鲁棒性角度检查训练集是否存在“语义相近但 label 相反”的可疑样本
- 使用字符级 TF-IDF + cosine similarity 筛选相似度 `>= 0.70` 且标签冲突的近重复样本对
- 输出人工核查表 `results/train_label_conflict_pairs_similarity_ge_0_70.csv`
- 对语义重复但标签不同的样本进行人工核查和清洗，更新 `train.csv`
- 在 `Project.md` 中记录数据投毒 / 错标排查方法、发现和处理结论

#### 对抗攻击鲁棒性分析
- 参与输入级字符扰动攻击测试设计，如 `o/O → 0`、`l/L → 1`、扩展 leetspeak 替换等
- 通过 Accuracy Drop、Flip Rate、Attack Success Rate 等指标评估现有模型在扰动输入下的脆弱性
- 相关结果用于报告中的安全分析和防护讨论

#### 人工抽查
- 从 val.csv 中随机选 30 条，人工核对「解释是否与标签一致、证据是否真实出现在文本中」。
- 记录不一致案例，反馈给钱/刘调优模型或 Prompt。

## 五、贺：系统集成、工程化与报告

- 批量推理输出 CSV，保留原始 `id`, `text`, `event`，并添加预测标签、置信度和解释。
